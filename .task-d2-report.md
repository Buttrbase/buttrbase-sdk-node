## What was done

1. **`jose` installed** as a production dependency (v5.x; zero transitive deps; ESM-first).

2. **`src/verify.ts` extended** (additive, no changes to existing exports):
   - Added `VerifierConfig` interface: `{ jwksUrl, issuer, audience? }` — mirrors Rust SDK's `VerifierConfig`.
   - Added `Verifier` class with `createRemoteJWKSet` (backed by `jose`):
     - `verifyToken(token): Promise<Claims>` — validates RS256 sig via JWKS kid lookup, enforces `iss` + expiry; enforces `aud` only when `audience` is set (matching Rust SDK `validate_aud = false` when `audience: None`).
     - `verifyBearer(authHeader): Promise<AuthContext>` — strips `Bearer `, calls `verifyToken`, maps to `AuthContext` via existing `claimsToAuthContext`.
     - `issuer` / `audience` read-only accessors.

3. **`src/index.ts`** — exports `Verifier` (value) and `VerifierConfig` (type).

4. **`tests/verifier.test.ts`** (20 tests, all offline):
   - Keypair generated via `jose.generateKeyPair('RS256')`, JWKS via `exportJWK` + `createLocalJWKSet`.
   - Tokens signed via `SignJWT`; Verifier's internal JWKS field monkey-patched with the local set (no network).
   - Positive: enriched Claims returned (roles, email, scopes, sub, org); no-audience mode; audience match.
   - Negative: tampered signature, corrupted signature, wrong issuer, expired token, wrong audience, missing aud when required, bad Bearer headers.

5. **CHANGELOG.md** — `## Unreleased` entry documenting the new `Verifier` API.
