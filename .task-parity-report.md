## Shape Divergences Handled

### OTP — email vs. phone (audit gap #1 and #2)

**Problem:** The existing `sendOtp(phone, appUuid)` and `verifyOtp(phone, code, appUuid)`
hit `/api/auth/otp` (phone-OTP), not the v1 email-OTP endpoint.

**Resolution:** Added `sendOtpV1(email, appUuid)` and `verifyOtpV1(email, otp, appUuid)`
targeting `/api/v1/auth/otp/send` and `/api/v1/auth/otp/verify` respectively.
The pre-existing phone methods are NOT repointed — that would break current callers.
The pre-existing `sendOtpEmail` / `verifyOtpEmail` already call the same v1 email
endpoint (they are existing aliases). `sendOtpV1` / `verifyOtpV1` are name-aligned
canonical forms added alongside them.

### Entitlements body field divergence (audit shape fix #1 and #2)

**Problem:** `entitlementsCheck(feature, orgUuid?)` sends `{ feature }` in the body
(not `{ feature_key }` as the Rust SDK does). `entitlementsCheckBatch` sends
`{ checks: [...] }` not `{ feature_keys: [...] }`.

**Resolution:** Added `checkEntitlement(featureKey)` sending `{ feature_key }` and
`checkEntitlements(featureKeys)` sending `{ feature_keys: string[] }`.
The pre-existing methods are retained unchanged — they are NOT silently repointed
(doing so would change the wire format and break existing integrations).

### Pricing untyped → typed (audit shape fix #3)

**Problem:** `pricingPreview/Quote/CheckoutSession` all accept `Record<string,unknown>`.

**Resolution:** Added typed variants `pricingPreviewTyped`, `pricingQuoteTyped`,
`checkoutSessionTyped` that accept `PricingPreviewRequest` / `CheckoutSessionRequest`
and return typed `PricingPreview` / `CheckoutSession`. Pre-existing untyped methods retained.

### usageReport untyped → typed (audit shape fix #4)

**Problem:** `usageReport(payload: Record<string,unknown>)` is untyped and uses bearer auth.

**Resolution:** Added `reportUsage(event: UsageEvent)` with typed `UsageEvent` struct.
It uses the SDK's current bearer (the client-credentials app bearer, which is analogous
to the Rust SDK's HTTP Basic app-level auth). Pre-existing `usageReport` retained.

### Analytics — missing `period` param (audit shape notes)

**Problem:** `analyticsAppOverview(appUuid)` and `analyticsOrgOverview(orgUuid)` don't
accept the `period` query param required by the Rust SDK's signatures.

**Resolution:** Added `appAnalyticsOverview(appUuid, period)` and
`orgAnalyticsOverview(orgUuid, period)` which include `?period=` in the URL.
Pre-existing methods retained.

### Wallet typed variant

`getWallet()` returns `Record<string,unknown>`. Added `walletSummary() → WalletSummary`.
Pre-existing `getWallet()` retained.

### billingHistory typed variant

`getBillingHistory()` returns `Record<string,unknown>`. Added `billingHistory() → Invoice[]`.
Pre-existing method retained.

### Teams typed variants

`listOrgTeams` / `getUserTeams` return `unknown[]`. Added `orgTeams` / `userTeams`
returning `TeamItem[]`. Pre-existing methods retained.

## Deferred / Not Implemented

### `environment()` / `isSandbox()` accessors

The Rust SDK auto-detects `live` vs `sandbox` from the `client_id` prefix and exposes
`environment()` / `is_sandbox()` / `base_url()` accessors. The Node SDK uses a single
constructor with a `baseUrl` option and defaults to the staging URL; there is no
prefix-based environment detection.

**Deferred:** Adding environment accessors would require changing the constructor
(or adding a new factory), which risks silent behavioral change. This is a
medium-effort construction-model divergence not covered by the "hard-missing method"
audit list, so it is deferred to a dedicated constructor-alignment task.

### `request_password_reset` / `reset_password` — already present

The audit initially listed these as possibly missing, but they are present in the
existing code (`requestPasswordReset` / `resetPassword` at the bottom of `client.ts`).
No action required.

### JWKS Verifier (`verify_token` / `verify_bearer` on the client object)

Marked as "in progress" in the audit by another agent. Not touched here.

### Org-level auth model note (auth for `report_usage`)

The Rust SDK's `report_usage` uses HTTP Basic auth (client_id:client_secret directly).
The Node SDK uses an OAuth2 bearer from the client-credentials grant instead (this is
the Node SDK's universal auth model — it does not do per-request HTTP Basic).
The `reportUsage` implementation therefore uses the SDK's bearer token, which is the
app-server bearer obtained from the client-credentials grant — semantically equivalent
(app-level, not user-level) but a different wire format. No divergence risk for
callers; noted here for completeness.
